Project

mailrails

This is a post to introduce a work in progress. Typed by hand and then passed through GPT-5.4 with the instruction to "make it more concise" and then reviewed for accuracy.

I wanted to use an agent to help clean up my Gmail inbox, but I didn't want to give it direct browser access, and Gmail's API scopes weren't granular enough for my workflow.

So I built mailrails, a proxy between the agent and Gmail:

me → agent → mailrails → Gmail

mailrails exposes only the Gmail actions I want the agent to use. Each action can be configured in one of two modes:

  • auto-execute (auto)
  • requires approval (ask me)

That lets the agent request sensitive actions without being able to execute them unchecked. If you connect Telegram, mailrails can notify you when an approval request is waiting.

Once you authorize mailrails to access your Gmail account, you connect your agent by giving it the getHelp endpoint and your mailrails API key.

Technical Notes

The hardest part was building the approval request flow.

When an approval request is shown to the user, it needs to clearly communicate what approving it will actually do. A human-readable endpoint name plus raw parameters is not enough.

The tricky case is endpoints that take arrays of entity IDs. For example, if the agent wants to call users.messages.batchModify, the user should not have to review a list of opaque Gmail message IDs. They need to see what those IDs refer to, such as the message subject and sender.

The straightforward solution is to make additional Gmail API calls to enrich those IDs before showing the request. In the batchModify case, that means calling users.messages.get for each message ID so the approval request can display something meaningful. The problem is this can very easily exceed Gmail API rate limits.

After some experimentation, I ended up adding a Gmail API request cache. That made these enrichment calls cheap enough to use freely.
